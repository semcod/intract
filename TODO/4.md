Tak — żeby **Intract** było powszechnie używane, nie powinno być tylko parserem komentarzy. Powinno stać się **warstwą kontraktów intencji dla różnych artefaktów software delivery**:

```text id="977n4n"
kod
funkcje
klasy
pliki
endpointy API
eventy
usługi
Dockerfile
docker-compose
Kubernetes
Terraform
CI/CD
Git hooks
testy
dokumentacja
artefakty release
```

Najważniejsza decyzja: **Intract ma być krótkim, przenośnym kontraktem intencji**, który można osadzić w wielu standardowych miejscach, a nie osobnym, zamkniętym językiem.

# 1. Docelowa idea

Obecnie masz:

```python id="vfu1ef"
# @intract.v1 scope:function intent:validate:user_permission priority:1 domain:security input:user,resource output:allowed effect:none forbid:write,network
```

Docelowo ten sam kontrakt powinien działać też w:

## OpenAPI

```yaml id="caf2sn"
paths:
  /users/{id}:
    get:
      x-intract:
        intent: read:user_profile
        priority: 1
        domain: user
        input: [user_id]
        output: [UserProfile]
        forbid: [write]
```

OpenAPI to formalny standard opisu HTTP API, więc Intract powinien używać rozszerzeń `x-intract`, zamiast wymyślać własny format API. ([openapis.org][1])

## AsyncAPI

```yaml id="kxq5lb"
channels:
  user.created:
    messages:
      UserCreated:
        x-intract:
          intent: publish:user_created_event
          priority: 2
          domain: events
          input: [User]
          output: [UserCreatedEvent]
          effect: [publish]
```

AsyncAPI opisuje message-driven APIs w formacie maszynowo czytelnym i jest naturalnym miejscem dla kontraktów intencji eventów. ([asyncapi.com][2])

## Dockerfile

```dockerfile id="ev1hwv"
# @intract.v1 scope:container intent:package:web_service priority:2 domain:runtime input:app_source output:oci_image effect:build forbid:secret_leak,root_user
FROM python:3.12-slim
```

## OCI image labels

```dockerfile id="861wbf"
LABEL io.intract.intent="package:web_service"
LABEL io.intract.scope="container"
LABEL io.intract.forbid="secret_leak,root_user"
```

Tu warto respektować standard OCI: klucze `org.opencontainers.image.*` są zarezerwowane dla OCI, więc Intract powinien używać własnej przestrzeni np. `io.intract.*`. ([https://opencontainers.github.io][3])

## GitHub Actions

```yaml id="lj8fgg"
# @intract.v1 scope:workflow intent:validate:pull_request priority:1 domain:ci input:source_tree output:quality_report effect:read forbid:publish
name: CI
```

## Kubernetes

```yaml id="d35m38"
metadata:
  annotations:
    intract.io/intent: "deploy:web_service"
    intract.io/scope: "service"
    intract.io/priority: "1"
    intract.io/forbid: "privileged,latest_image"
```

## SARIF output

Intract powinien eksportować naruszenia do SARIF, bo SARIF jest standardowym formatem wyników narzędzi analizy statycznej. To pozwala pokazywać wyniki w Code Scanning / PR review / narzędziach security. ([OASIS Open][4])

# 4. System pluginów

Najważniejszy krok: Intract powinien mieć plugin registry.

## Typy pluginów

```text id="qoas3n"
ParserPlugin       → wyciąga kontrakty z artefaktu
ValidatorPlugin    → waliduje kontrakt
ReporterPlugin     → zapisuje wynik
IntegrationPlugin  → łączy Intract z zewnętrznym narzędziem
PolicyPlugin       → definiuje zasady fail/warn
```

## Interfejs bazowy

```python id="g9wjo2"
from typing import Protocol


class IntractPlugin(Protocol):
    name: str
    version: str

    def supports(self, artifact) -> bool:
        ...

    def run(self, context):
        ...
```

## Parser plugin

```python id="vd2i8h"
class ParserPlugin(Protocol):
    name: str

    def supports_path(self, path: str) -> bool:
        ...

    def parse(self, artifact) -> list[ContractRecord]:
        ...
```

## Validator plugin

```python id="2ltw4h"
class ValidatorPlugin(Protocol):
    name: str

    def supports_contract(self, contract) -> bool:
        ...

    def validate(self, contract, artifact, context) -> ValidationResult:
        ...
```

## Reporter plugin

```python id="x5sj8n"
class ReporterPlugin(Protocol):
    name: str
    extension: str

    def render(self, report) -> str:
        ...
```

# 8. Kontrakty dla endpointów

## Inline FastAPI

```python id="t04874"
# @intract.v1 scope:endpoint intent:read:user_profile priority:1 domain:user input:user_id output:UserProfile effect:read forbid:write require:authorize.user_access
@app.get("/users/{user_id}")
def get_user(user_id: str):
    ...
```

## OpenAPI

```yaml id="9yz645"
paths:
  /users/{user_id}:
    get:
      x-intract:
        scope: endpoint
        intent: read:user_profile
        priority: 1
        domain: user
        input: [user_id]
        output: [UserProfile]
        effect: [read]
        forbid: [write]
        require:
          - authorize.user_access
```

## Validator endpointów

Sprawdza:

```text id="0uravv"
1. Czy path/method zgadza się z intent.
2. Czy inputy są w path/query/body.
3. Czy output jest w response schema.
4. Czy require auth jest spełnione.
5. Czy forbid:write nie jest złamane w GET.
6. Czy effect zgadza się z metodą HTTP.
```

Reguły:

```text id="tqxwru"
GET     → default effect: read
POST    → default effect: create/write
PUT     → default effect: replace/write
PATCH   → default effect: update/write
DELETE  → default effect: delete/write
```

## AsyncAPI

```yaml id="yhf62d"
channels:

          scope: event

          input: [User]

          effect: [publish]
          require:
            - validate.user_payload
```

Validator eventów sprawdza:

```text id="l28mis"
- czy kanał/event pasuje do intent,
- czy message schema istnieje,
- czy wymagane require są pokryte,
- czy producer/consumer jest oznaczony,
- czy event nie łamie forbid, np. forbid:pii.
```

# 10. Kontrakty dla Docker i kontenerów

## Dockerfile

```dockerfile id="x29qs4"
# @intract.v1 scope:container intent:package:api_service priority:1 domain:runtime input:app_source output:oci_image effect:build forbid:secret_leak,root_user,latest_tag validate:no_forbidden_effect,container_security
FROM python:3.12-slim

USER app
```

Validator kontenera:

```text id="h7tmwv"
- brak USER root,
- brak secrets w ENV,
- brak ADD z URL,
- brak latest tag,
- expose port zgodny z manifestem,
- healthcheck obecny jeśli require:healthcheck.
```

## docker-compose

```yaml id="5d5lra"
services:
  api:
    image: my-api:1.0.0
    labels:
      io.intract.scope: service
      io.intract.intent: serve:http_api
      io.intract.priority: "1"
      io.intract.require: connect.database,expose.healthcheck
      io.intract.forbid: privileged,host_network
```

Validator usług:

```text id="typr6a"
- czy wymagane zależności istnieją,
- czy ports są jawne,
- czy privileged=false,
- czy healthcheck jest obecny,
- czy env nie zawiera sekretów.
```

# 12. Kontrakty dla CI/CD

## GitHub Actions


Validator CI/CD:

```text id="10gtg7"
- PR workflow nie publikuje paczek,
- push-to-main może publish,

- secrets nie są printowane,
- deploy wymaga testów,
- workflow ma minimal permissions.
```

```yaml id="0d88j1"
  - test
```

# 17. `intent.yaml` jako centrum projektu

Przykładowy projektowy `intent.yaml`:

```yaml id="17ilpi"
version: intract.v1

project:
  name: my-service
  intent: serve:user_api

contracts:
  - id: project.service
    scope: project
    intent: serve:user_api
    priority: 1
    domain: service
    input: [http_request]
    output: [http_response]
    require:
      - expose.user_endpoints
      - validate.user_input
      - authorize.user_access
      - package.docker_image
      - deploy.kubernetes_service
      - validate.ci_pipeline
    forbid:
      - secret_leak
      - unauthenticated_write

artifacts:
  openapi.yaml:
    kind: openapi
    provides:
      - expose.user_endpoints

  Dockerfile:
    kind: dockerfile
    provides:
      - package.docker_image
    forbid:
      - root_user
      - latest_tag
      - secret_leak

  .github/workflows/ci.yml:
    kind: github_actions
    provides:
      - validate.ci_pipeline
    require:
      - run.tests
      - run.lint
      - run.intract

  k8s/deployment.yaml:
    kind: kubernetes
    provides:
      - deploy.kubernetes_service
    require:
      - expose.healthcheck
      - configure.resources
```

To jest najważniejsza funkcja, jeśli Intract ma być używany szeroko: **manifest wiąże kod, API, kontener, deploy i CI w jeden graf intencji**.

# 21. Jak sprawić, żeby Intract był „respektowany”

Nie wystarczy mieć dobry parser. Trzeba zrobić 5 rzeczy:

## 1. Użyć istniejących standardów

```text id="f7nubn"
OpenAPI    → x-intract
AsyncAPI   → x-intract
Docker/OCI → io.intract.* labels
Kubernetes → intract.io/* annotations
CI/CD      → GitHub Action / GitLab template
SARIF      → wyniki walidacji
```

## 2. Zapewnić JSON Schema

Każdy `intent.yaml` powinien być walidowalny przez standardowe narzędzia YAML/JSON.

## 3. Dodać pre-commit i CI

Jeśli Intract nie działa w commit/PR, nie będzie realnie używany.

## 4. Dodać raporty dla ludzi i maszyn

```text id="jlpgs8"
Markdown dla ludzi
JSON dla narzędzi
SARIF dla code scanning
Mermaid dla architektury
```

## 5. Utrzymać prostą składnię

Najważniejsze, żeby nadal dało się napisać:

```python id="m5f0dg"
# @intract validate:user_permission p1 no:network
```

oraz pełną wersję:

```python id="5w8kqs"
# @intract.v1 scope:function intent:validate:user_permission priority:1 domain:security input:user,resource output:allowed forbid:network meaning:"check permission"
```

# 22. Najważniejsza rekomendacja

Zrób z Intract **standard krótkich kontraktów intencji dla artefaktów**, nie tylko narzędzie do kodu.

Minimalny kierunek:

```text id="zjgq83"
Intract Core
  parser + signature + validation

Intract Plugins
  code + openapi + docker + ci + k8s

Intract Reports
  json + markdown + sarif + mermaid

Intract Integrations
  pre-commit + GitHub Action + vallm + reDUP
```

Wtedy Intract może być używany jako:

```text id="35zfk8"
linter intencji,
walidator CI,
warstwa dokumentacji,
graf architektury,
kontrola kontraktów endpointów,
kontrola kontraktów deploymentu,
detektor duplikacji intencji w reDUP,
```
